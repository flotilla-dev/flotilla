# Runtime I/O Specification (v1.4-draft)

## 1. Executive Summary

Runtime I/O is the behavioral contract for the canonical input and output objects used by `FlotillaRuntime`.

It defines:

- `RuntimeRequest` - canonical input consumed by `FlotillaRuntime.run()` and `FlotillaRuntime.stream()`
- `RuntimeResponse` - canonical synchronous terminal result generated by `FlotillaRuntime.run()`
- `RuntimeEvent` - canonical streaming output generated by `FlotillaRuntime.stream()`
- `ResumeToken` - opaque suspend/resume identifier shared across all three I/O objects

These objects are transport-agnostic, JSON-serializable, execution-phase scoped, and independent of HTTP specifics. They encode runtime I/O semantics only.

## 2. Architectural Context

Runtime I/O sits at the external boundary of `FlotillaRuntime`.

Upstream callers construct `RuntimeRequest` objects and pass them to runtime entry points:

- `FlotillaRuntime.run(request: RuntimeRequest) -> RuntimeResponse`
- `FlotillaRuntime.stream(request: RuntimeRequest) -> AsyncIterator[RuntimeEvent]`

Downstream, `FlotillaRuntime` maps runtime execution outcomes, durable thread entries, policy decisions, and agent events into either a single `RuntimeResponse` or a stream of `RuntimeEvent` objects.

Runtime I/O depends on:

- `ContentPart` for all user-visible input and output content
- `Thread Model` for durable execution outcomes
- `FlotillaRuntime` for lifecycle enforcement
- `ResumeAuthorizationPolicy` for resume permission semantics

Runtime I/O does not define storage, orchestration, lifecycle enforcement, or transport behavior. It defines only the canonical shape and meaning of values crossing the runtime boundary.

## 3. Core Concepts

### RuntimeRequest

`RuntimeRequest` represents a single execution phase initiation attempt. Each valid request is consumed by either `run()` or `stream()`.

Structural schema:

```json
{
  "thread_id": "string",
  "user_id": "string",
  "timestamp": "ISO-8601 string",
  "correlation_id": "string | null",
  "trace_id": "string | null",
  "resume_token": "string | null",
  "content": [ "ContentPart" ],
  "transport_metadata": { "...": "..." }
}
```

### RuntimeResponse

`RuntimeResponse` represents the terminal result of a synchronous runtime invocation. Exactly one response is returned by `run()`.

Structural schema:

```json
{
  "type": "enum",
  "phase_id": "string",
  "thread_id": "string",
  "timestamp": "ISO-8601 string",
  "correlation_id": "string | null",
  "trace_id": "string | null",
  "content": [ "ContentPart" ],
  "resume_token": "string | null"
}
```

Valid `type` values are:

- `COMPLETE`
- `SUSPEND`
- `ERROR`

### RuntimeEvent

`RuntimeEvent` represents streaming runtime output. Events are yielded by `stream()` as an `AsyncIterator`.

Structural schema:

```json
{
  "type": "enum",
  "phase_id": "string",
  "thread_id": "string",
  "timestamp": "ISO-8601 string",
  "correlation_id": "string | null",
  "trace_id": "string | null",
  "content": [ "ContentPart" ] | null,
  "resume_token": "string | null"
}
```

Valid non-terminal event types are:

- `START`
- `PART`

Valid terminal event types are:

- `COMPLETE`
- `SUSPEND`
- `ERROR`

### ResumeToken

`ResumeToken` is an opaque string used to identify a suspend/resume pattern across requests, synchronous responses, and streaming events.

Conceptually, a token is derived from durable suspend state:

```json
{
  "thread_id": "string",
  "phase_id": "string",
  "suspend_entry_id": "string",
  "issued_at": "ISO-8601 string",
  "expires_at": "ISO-8601 string"
}
```

Implementations MAY encode a resume token as a signed token, Base64 JSON, or opaque ID referencing durable state. Callers MUST treat it as opaque.

## 4. Responsibilities

Runtime I/O MUST:

- Define the canonical input object for runtime execution.
- Define the canonical synchronous output object for terminal execution results.
- Define the canonical streaming output object for execution progress and terminal outcomes.
- Provide a common `ResumeToken` representation across request, response, and event flows.
- Express all user-visible content through `ContentPart`.
- Preserve correlation and tracing identifiers when present.
- Remain transport-agnostic and JSON-serializable.

## 5. Non-Responsibilities

Runtime I/O MUST NOT define:

- Business logic
- Agent orchestration
- Storage semantics
- Durable append rules
- CAS enforcement
- Thread concurrency control
- Runtime lifecycle enforcement
- Transport-specific behavior
- HTTP status codes, headers, or routing

Execution lifecycle rules, durable append semantics, CAS enforcement, terminal state guarantees, and resume authorization are defined by the `FlotillaRuntime` and related policy specifications.

## 6. Behavioral Contract

### RuntimeRequest Field Semantics

`thread_id` is REQUIRED. It identifies the thread to execute against. Runtime MUST fail if the thread does not exist. Runtime MUST NOT create the thread from a `RuntimeRequest`.

`user_id` is REQUIRED. It identifies the requesting user for auditing and policy decisions.

`timestamp` is OPTIONAL on construction and defaults to the current UTC time. It records request creation time for logging only. Runtime MUST NOT use it for timeout enforcement or durable ordering. Durable store timestamps remain authoritative.

`correlation_id` is OPTIONAL. Runtime MUST pass it through unchanged when present. Runtime MUST NOT require it.

`trace_id` is OPTIONAL. Runtime MAY use it for tracing. Runtime MUST NOT require it.

`resume_token` is OPTIONAL. If present, the request represents a resume attempt. Runtime MUST validate the token against durable thread state and MUST reject invalid or expired tokens. If absent, the request represents a new phase initiation.

`content` is REQUIRED and MUST contain at least one `ContentPart`. It contains user-provided input content.

`transport_metadata` is OPTIONAL. Runtime MUST treat it as opaque and MUST NOT depend on it for correctness.

### RuntimeResponse Semantics

`phase_id` is REQUIRED. It identifies the execution phase that produced the terminal synchronous result.

`timestamp` is OPTIONAL on construction and defaults to the current UTC time.

`COMPLETE` represents successful synchronous execution. It corresponds to durable `AgentOutput`. `content` MUST contain final agent output. `resume_token` MUST be null.

`SUSPEND` represents suspended synchronous execution. It corresponds to durable `SuspendEntry`. `content` MUST explain suspension state and next steps. `resume_token` MUST be present.

`ERROR` represents execution failure or request rejection. It corresponds to durable `ErrorEntry` or pre-initiation rejection. `content` MUST contain user-facing error information. `resume_token` MUST be null.

Exactly one `RuntimeResponse` MUST be returned by each `run()` invocation.

### RuntimeEvent Semantics

`phase_id` is REQUIRED. It identifies the execution phase that emitted the event.

`timestamp` is OPTIONAL on construction and defaults to the current UTC time.

`START` indicates the beginning of streaming execution output.

`PART` represents incremental streaming content. It typically contains a single `ContentPart` of type `TEXT`. It MAY repeat multiple times.

`COMPLETE` is terminal and represents successful execution. It corresponds to durable `AgentOutput`. It includes final content when available. It MUST NOT include `resume_token`.

`SUSPEND` is terminal and represents suspended execution. It corresponds to durable `SuspendEntry`. It includes `content` when available. It MUST include `resume_token`.

`ERROR` is terminal and represents execution failure or request rejection. It corresponds to durable `ErrorEntry` or pre-initiation rejection. It includes `content` when available. It MUST NOT include `resume_token`.

Streaming invocations MUST emit exactly one terminal event.

### ResumeToken Semantics

`ResumeToken` MUST:

- Be derived from durable `SuspendEntry`.
- Be validated against durable thread state.
- Become invalid if expired, mismatched to thread, mismatched to the current suspend tail, or mismatched to `suspend_entry_id`.
- Be treated as opaque by callers.

`ResumeToken` MUST NOT contain `ContentPart`.

Resume token validation alone does not guarantee resume permission. Runtime MUST additionally enforce resume authorization via `ResumeAuthorizationPolicy`.

## 7. State Model

Runtime I/O objects are immutable value objects.

Runtime I/O does not own durable state. Durable thread state is owned by `ThreadEntryStore` and interpreted through the `Thread Model`.

`ResumeToken` identifies durable suspend state but does not itself own that state. Token validity is determined by runtime validation against the durable thread state and runtime-specific token rules.

Runtime I/O fields MUST NOT be mutated by `run()` or `stream()`. Runtime MAY construct new response and event objects derived from request metadata, durable entries, policy results, and agent output.

## 8. Interaction Model

Synchronous interaction:

1. Caller submits a `RuntimeRequest` to `FlotillaRuntime.run()`.
2. Runtime validates the request and executes or rejects the phase.
3. Runtime returns exactly one terminal `RuntimeResponse`.

Streaming interaction:

1. Caller submits a `RuntimeRequest` to `FlotillaRuntime.stream()`.
2. Runtime validates the request and starts the phase.
3. Runtime yields ordered `RuntimeEvent` objects.
4. Runtime yields exactly one terminal event.
5. No events are yielded after the terminal event.

Suspend/resume interaction:

1. A suspended runtime result includes a `ResumeToken`.
2. Caller includes that token in a subsequent `RuntimeRequest`.
3. Runtime validates the token and resume authorization.
4. Runtime either resumes execution or returns/reports an error.

## 9. Configuration Contract

Runtime I/O has no direct configuration inputs.

Runtime behavior affecting I/O MAY be configured by collaborating components, including:

- `FlotillaRuntime`
- `ResumeAuthorizationPolicy`
- token encoding and validation mechanisms
- telemetry and tracing integrations

Those collaborators MUST NOT change the structural contract of `RuntimeRequest`, `RuntimeResponse`, `RuntimeEvent`, or `ResumeToken`.

## 10. Extension Points

Runtime I/O may be extended only through explicitly opaque fields or implementation-owned token encoding.

Supported extension points:

- `transport_metadata` on `RuntimeRequest`
- `correlation_id`
- `trace_id`
- implementation-specific `ResumeToken` encoding

Unsupported extensions:

- Adding ad hoc response payload channels outside `ContentPart`
- Adding transport-specific requirements to canonical I/O objects
- Embedding durable storage details in runtime I/O
- Introducing new terminal states without updating this specification

## 11. Constraints & Guarantees

Runtime I/O guarantees:

- Transport independence
- JSON-serializable object shapes
- Deterministic execution-phase scoping
- Explicit synchronous terminal states
- Streaming parity with synchronous outcomes
- Strict separation from storage semantics
- No embedded concurrency or lifecycle rules
- Opaque resume token handling

Runtime events MUST be emitted in causal order.

Exactly one terminal `RuntimeEvent` MUST be emitted by `stream()`.

No `RuntimeEvent` may follow a terminal event.

`resume_token` MUST appear only on `SUSPEND` responses or events.

Terminal responses MUST include `content`. Runtime events MAY omit `content` for non-content-bearing events.

## 12. Observability

Runtime I/O supports observability by carrying:

- `correlation_id` for application-level request correlation
- `trace_id` for tracing integrations
- `timestamp` fields for diagnostic visibility
- `user_id` for audit-relevant request context

Runtime I/O does not define logging, metrics, spans, or audit sinks. Those behaviors are owned by `TelemetryPolicy`, transport adapters, and runtime collaborators.

Observability fields MUST NOT be required for correctness, ordering, timeout enforcement, or durable identity.
